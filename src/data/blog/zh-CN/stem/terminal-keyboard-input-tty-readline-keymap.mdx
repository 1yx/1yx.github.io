终端键盘输入在到达应用程序之前会流经三个层级：底层的 ASCII 控制字符编码、中间的 **TTY** 驱动（通过 termios 配置）以及顶层的应用程序处理（GNU Readline, <span title="Friendly Interactive SHell — 用户友好的交互式 shell">Fish</span>, Claude Code）。理解这个堆栈结构可以解释为什么 `Ctrl+M` 与 `Enter` 相同，为什么大多数工具中无法重新绑定 `Ctrl+C`，以及为什么 <span title="Friendly Interactive SHell — 用户友好的交互式 shell">Fish</span> 实现了自己的行编辑器而不是使用 Readline。

<KeyTakeaways>
- 终端输入处理分为三层：ASCII 编码 → **TTY** 驱动 (termios) → 应用程序 (Readline 或自定义)
- `Ctrl+M` 和 `Enter` 在 ASCII 层面产生相同的字节 (`0x0D`) —— 这是 1960 年代的标准，而非工具配置
- 标准模式 (Canonical Mode) 的行编辑是为 **TTY** (Teletypewriter) 设计的，因为纸张上的内容无法擦除
- GNU Readline 键绑定是源自 Emacs 惯例的业界标准，而非任何 POSIX 规范定义
- 现代工具（<span title="Friendly Interactive SHell — 用户友好的交互式 shell">Fish</span>, Claude Code）绕过了 Readline，但保留了 Emacs 键绑定作为肌肉记忆惯例
</KeyTakeaways>

## 终端输入流水线：从键盘到应用程序

当你在 Ghostty、Kitty 或 Alacritty 等终端模拟器中按下按键时，字节会经过一个定义明确的流水线：

```
终端模拟器 (Ghostty)     →  捕获键盘事件，将字节写入 PTY Master
        ↓
TTY 驱动 (内核)           →  PTY Slave 端，根据 termios 设置拦截或传递字节
        ↓
应用程序 (Fish 等)        →  在 Raw Mode 下，自行处理所有按键
```

终端模拟器是现代版的 **TTY**。PTY (Pseudo-TTY) 对是现代版的串行电缆。**TTY** 驱动则是 1970 年代处理 **TTY** 串行数据的同一个内核模块。

## 1. **TTY** 与 Terminal：名称的历史

**TTY** 代表 TeleTYpewriter（电传打字机）—— 一种带有键盘和打印头的物理机电设备，通过串行电缆连接到计算机。Terminal（终端）是其后续概念。早期的终端也是硬件（如 DEC VT100），用 CRT 屏幕取代了纸张。

### 硬件演进

| 时代 | 设备 | 接口 | 显示 |
|-----|------|------|------|
| 1960s | **TTY**（物理电传打字机） | 串行端口 | 纸张 |
| 1970s | 视频终端 (VT100) | 串行端口 | CRT 屏幕 |
| 1980s | 终端模拟器 (xterm) | PTY (Pseudo-TTY) | X11 窗口 |
| 2020s | Ghostty / Kitty / Alacritty | PTY | GPU 加速窗口 |

内核的 **TTY** 驱动和 PTY 接口是为 **TTY** 设计的，并一直沿用至今。今天，Ghostty 是”模拟的 **TTY**”，PTY 是”串行电缆”，而 **TTY** 驱动是”处理串行数据的内核模块”。

### 术语渊源

| 名称 | 原始含义 | 现代含义 |
|------|----------|----------|
| **TTY** | 电传打字机 (Teletypewriter) | 内核终端驱动模块 |
| PTY | 伪电传打字机 (Pseudo-TTY) | 软件模拟的 **TTY**（Master/Slave 对） |
| terminal | 物理终端设备 | 终端模拟器软件 (Ghostty) |
| console | 系统控制台 | TUI 或系统主终端 |
| `stty` | Set teletype | 配置 **TTY** 驱动参数 |
| `/dev/tty` | **TTY** 设备文件 | 当前终端的设备文件 |

## 2. ASCII 控制字符编码

Ctrl 组合键的编码规则定义在 1960 年代的 ASCII 标准中：

```
Ctrl + key = key & 0x1F  (清除第 5 和第 6 位，即与 0b00011111 进行 AND 运算)
```

该规则适用于 ASCII 范围 `0x40`–`0x5F` 的字符（大写字母 `A`–`Z` 以及符号 `@`, `[`, `\`, `]`, `^`, `_`）。它不适用于小写字母或该范围之外的字符。

| 按键 | 计算方式 | 字节 | ASCII 名称 | 含义 |
|-----|---------|------|-----------|------|
| `Ctrl+M` | 0x4D & 0x1F = 0x0D | `0x0D` | CR | 等同于 Enter |
| `Ctrl+J` | 0x4A & 0x1F = 0x0A | `0x0A` | LF | 换行 (Line Feed) |
| `Ctrl+I` | 0x49 & 0x1F = 0x09 | `0x09` | HT | 等同于 Tab |
| `Ctrl+[` | 0x5B & 0x1F = 0x1B | `0x1B` | ESC | Escape |
| `Ctrl+C` | 0x43 & 0x1F = 0x03 | `0x03` | ETX | 文本结束 (End of text) |
| `Ctrl+D` | 0x44 & 0x1F = 0x04 | `0x04` | EOT | 传输结束 (End of transmission) |

`Ctrl+M = Enter` 和 `Ctrl+I = Tab` 不是工具配置。在字节层面，它们是同一个值。这是由 ASCII 标准 (ANSI X3.4-1963) 定义的，任何应用程序都无法更改。

## 3. **TTY** 驱动 (termios)

**TTY** 驱动是位于终端和进程之间的内核级软件层。POSIX (IEEE 1003.1 / ISO/IEC 9945) 通过 `termios` 结构定义了其行为：

```c
struct termios {
    tcflag_t c_iflag;    // 输入模式标志
    tcflag_t c_oflag;    // 输出模式标志
    tcflag_t c_cflag;    // 控制模式标志
    tcflag_t c_lflag;    // 本地模式标志 (echo, icanon, ...)
    cc_t     c_cc[NCCS]; // 特殊字符数组
};
```

### `c_cc` 数组 (Linux, 19 个条目)

| 索引 | stty 名称 | 默认值 | 含义 |
|-----|-----------|-------|------|
| VINTR | `intr` | <span title="^（caret）代表 Ctrl 修饰符">`^C`</span> | 发送 SIGINT |
| VQUIT | `quit` | <span title="^（caret）代表 Ctrl 修饰符">`^\`</span> | 发送 SIGQUIT |
| VERASE | `erase` | <span title="^? 是 stty 的一个特殊惯例，用 ^ 加 ? 来表示 DEL">`^?`</span> | 删除前一个字符 |
| VKILL | `kill` | <span title="^（caret）代表 Ctrl 修饰符">`^U`</span> | 删除整行 |
| VEOF | `eof` | <span title="^（caret）代表 Ctrl 修饰符">`^D`</span> | EOF 标记 / 刷新缓冲区 |
| VSTART | `start` | `^Q` | 恢复输出 (XON) |
| VSTOP | `stop` | `^S` | 暂停输出 (XOFF) |
| VSUSP | `susp` | <span title="^（caret）代表 Ctrl 修饰符">`^Z`</span> | 发送 SIGTSTP |
| VEOL | `eol` | `<undef>` | 行结束字符 |
| VMIN | `min` | `1` | `read()` 最小字符数 (仅限非标准模式) |
| VTIME | `time` | `0` | `read()` 超时时间 (仅限非标准模式) |

在 POSIX `termios` 结构中，标准模式 (Canonical Mode) 和非标准模式是互斥的，因此驱动程序会复用 `c_cc` 数组中的内存槽位。`VMIN` 通常与 `VEOF` (<span title="^（caret）代表 Ctrl 修饰符">`^D`</span>) 共享同一个索引，而 `VTIME` 与 `VEOL` 共享。这就是为什么系统程序员在切换模式时必须格外小心。

### `stty` 命令

`stty` (Set Teletype) 是用于配置 termios 结构的底层工具。它在后台调用 `tcgetattr()` 和 `tcsetattr()`：

```
$ stty -a
intr = ^C     →  SIGINT (中断)
quit = ^\     →  SIGQUIT (退出 + Core Dump)
susp = ^Z     →  SIGTSTP (挂起)
erase = ^?    →  删除前一个字符
kill = ^U     →  删除整行
eof = ^D      →  刷新缓冲区 / EOF 标记
```

注意，`Ctrl+D` (VEOF) 并不是一个信号。在标准模式下，它指示 **TTY** 驱动立即将当前缓冲区内的所有字节返回给等待的 `read()` 系统调用。如果缓冲区为空（即在全新的一行直接按下 `Ctrl+D`），`read()` 会返回 0 字节。在 POSIX 系统中，`read()` 读到 0 字节被应用程序约定俗成地解释为 End-of-File (EOF)。

## 4. 标准模式：为 **TTY** 设计的行编辑

标准模式 (**Cooked Mode**，也称为 Canonical Mode) 是 **TTY** 驱动的默认行编辑模式。**TTY** 驱动代表用户维护一个内部输入缓冲区 —— 只有在按下 Enter 后，应用程序才会收到完整的行。

这种设计是为了 **TTY** 而构建的，因为纸张上的内容无法擦除。编辑只能发生在 **TTY** 驱动的内部缓冲区中。纸张上显示的内容是单独的、不可逆的记录。

在 1960 年代早期的 **TTY** 上，擦除字符是 <span title="^（caret）代表 Ctrl 修饰符">`^H`</span> (BS, `0x08`)，纸上标记删除的惯例是打印 `#`。<span title="^? 是 stty 的一个特殊惯例，用 ^ 加 ? 来表示 DEL">`^?`</span> (DEL, `0x7F`) 后来成为了标准的擦除字符，由 1970 年代的 DEC VT 系列终端确立。

<span title="^? 是 stty 的一个特殊惯例，用 ^ 加 ? 来表示 DEL">`^?`</span> (DEL) 的起源是物理层面的：在 打孔纸带 (Punched paper tape) 时代，字符是通过在纸带上打孔来记录的。一旦孔打好了，就无法“取消打孔”。`0x7F` 的二进制表示是 `1111111`（7 个孔全部打满）。如果操作员打错了字符，他们只需倒退纸带并打满所有孔（即“擦除 (Rubout)”字符），机器读取时会直接忽略这个全孔字符。这种物理限制正是 `DEL` 成为删除标准的原因。

现代 Linux 系统默认使用 <span title="^? 是 stty 的一个特殊惯例，用 ^ 加 ? 来表示 DEL">`^?`</span> 作为 `VERASE`，而一些较旧的 BSD 衍生系统仍默认使用 <span title="^（caret）代表 Ctrl 修饰符">`^H`</span>。

| 按键 | **TTY** 缓冲区操作 | 纸上的效果 (1960s **TTY** 时代) |
|-----|---------------|---------------------------|
| <span title="^（caret）代表 Ctrl 修饰符">`^H`</span> (早期擦除) | 删除最后一个字符 | 退回一个位置，打印 `#` 标记删除 |
| <span title="^（caret）代表 Ctrl 修饰符">`^W`</span> (werase) | 删除最后一个单词 | 逐个覆盖字符 |
| <span title="^（caret）代表 Ctrl 修饰符">`^U`</span> (kill) | 清除整行 | 打印 `@` 标记该行作废，换行重打 |
| <span title="^（caret）代表 Ctrl 修饰符">`^R`</span> (reprint) | 重新输出缓冲区内容 | 重新打印整洁的当前缓冲区 |

考虑这个输入序列：`fix the bu` + <span title="^（caret）代表 Ctrl 修饰符">`^W`</span> + <span title="^（caret）代表 Ctrl 修饰符">`^W`</span> + `the bug`

纸上显示的内容（带有划掉标记）：

```
fix the bu####the bug
         ^^^^
         两个 ^W 划掉了 "bu" 和 "the"
```

**TTY** 缓冲区（操作员不可见，逐步演变）：

```
fix the bu          ← 输入到这里
fix the             ← ^W 删除了 "bu"
fix                 ← ^W 删除了 "the"
fix the bug         ← 输入 "the bug"
```

纸张和缓冲区是不同步的。纸张显示了带有划掉标记的完整历史。缓冲区只持有整洁的当前状态。这就是为什么存在 <span title="^（caret）代表 Ctrl 修饰符">`^R`</span> (reprint) —— 操作员忘记了缓冲区里到底是什么，需要重新打印一份整洁的内容。

当支持光标移动和屏幕擦除功能的视频终端 (VT100) 出现时，**TTY** 驱动可以发送 `BS SPACE BS` 序列来实际擦除屏幕上的字符。纸张和缓冲区终于在视觉上同步了。现代终端模拟器 (Ghostty) 继承了这一机制。

## 5. **TTY** 拦截 vs 应用程序处理

**TTY** 驱动拦截它识别的字节并自行处理（发送信号、编辑行）。它不识别的字节则传递给应用程序：

```
键盘 Ctrl+C → 0x03 → TTY 拦截 → 发送 SIGINT → 进程收到信号
键盘 Ctrl+A → 0x01 → TTY 传递 → 到达 Readline/Fish → 应用程序处理
```

这意味着：

- `Ctrl+C` (SIGINT), `Ctrl+Z` (SIGTSTP), `Ctrl+\` (SIGQUIT) —— 由 **TTY** 层处理
- `Ctrl+A` (跳到行首), `Ctrl+E` (跳到行尾), `Ctrl+K` (删除至行尾) —— 由 Readline 或应用程序层处理
- `Ctrl+U`, `Ctrl+W`, `Ctrl+D` —— 在标准模式下由 **TTY** 处理，但通常被 Readline 等应用程序重新定义（例如 `Ctrl+U` 变成“删除至行首”而非“删除整行”）

## 6. Readline 与 POSIX：什么是标准，什么不是

Readline 并不是一个标准。它是 GNU Readline —— 最初由 Brian Fox 为 GNU 项目编写的行编辑库（约 1988-1989 年随 <span title="Bourne Again SHell — 大多数 Linux 发行版的默认 shell">Bash</span> 发布）。其键绑定源自 Emacs 惯例，而非任何正式规范。

### 历史时间线

| 年份 | 里程碑 | 编辑能力 |
|------|-------|---------|
| 1970s | Unix `sh` | 仅限 termios **Cooked Mode** (<span title="^（caret）代表 Ctrl 修饰符">`^U`</span> 删行, <span title="^（caret）代表 Ctrl 修饰符">`^W`</span> 删词)。无光标移动。 |
| 1978 | BSD <span title="C Shell — 类似 Bash 的 Unix shell">csh</span> | Bill Joy 引入历史替换 (`!!`, `!$`)，但仍无交互式行编辑 |
| ~1981 | <span title="TENEX C Shell — csh 的增强版">tcsh</span> | Ken Greer 为 <span title="C Shell — 类似 Bash 的 Unix shell">csh</span> 添加 Emacs 风格交互编辑 (`Ctrl+A` Home, `Ctrl+E` End) |
| ~1983 | <span title="KornShell — AT&T 的 David Korn 开发的 Unix shell">ksh</span> (AT&T) | David Korn 独立添加 Emacs 模式和 Vi 模式行编辑 |
| 1988–89 | GNU Readline | Brian Fox 为 <span title="Bourne Again SHell — 大多数 Linux 发行版的默认 shell">Bash</span> 创建可复用的库。采纳了 Emacs 惯例。 |
| 1990s | POSIX | 标准化了 termios 接口。未标准化行编辑键绑定。 |

### 标准化鸿沟

```
POSIX termios          — TTY 驱动行为 (标准化)
    ↑ 没有任何标准涵盖 ↑
BSD tcsh / GNU Readline — 行编辑键绑定 (事实标准，非正式规范)
    ↑ 每个工具自行实现 ↑
Fish / Helix / Claude Code — 不使用 GNU Readline；独立实现行编辑
```

关键点：

- POSIX 未定义 `Ctrl+A` = 行首或 `Ctrl+E` = 行尾。POSIX 仅指定了 `VERASE`、`VKILL` 和 `VWERASE` 等 `c_cc` 条目的语义。
- Emacs 键绑定是事实标准，通过 <span title="TENEX C Shell — csh 的增强版">tcsh</span>/<span title="KornShell — AT&T 的 David Korn 开发的 Unix shell">ksh</span> → GNU Readline → <span title="Bourne Again SHell — 大多数 Linux 发行版的默认 shell">Bash</span> 传播。没有任何规范定义它们。
- <span title="Friendly Interactive SHell — 用户友好的交互式 shell">Fish</span> 不使用 GNU Readline。它实现了自己的行编辑器，但遵循相同的 Emacs 键位，因为这些已经成为了肌肉记忆。

## 7. **Raw Mode**：当应用程序接管一切

### **Cooked Mode** vs **Raw Mode**

**Raw Mode** 是一种 termios 配置，它禁用所有 **TTY** 驱动处理，让应用程序直接接收原始字节。

| 行为 | **Cooked Mode** (默认) | **Raw Mode** |
|------|-------------------|---------|
| 行缓冲 | 按下 Enter 后交付输入 | 每个字节立即交付 |
| <span title="^（caret）代表 Ctrl 修饰符">`^C`</span> → SIGINT | **TTY** 拦截，发送信号 | 字节 `0x03` 直接传给应用程序 |
| <span title="^（caret）代表 Ctrl 修饰符">`^U`</span> / <span title="^（caret）代表 Ctrl 修饰符">`^W`</span> | **TTY** 执行行/词删除 | 字节直接传给应用程序 |
| <span title="^（caret）代表 Ctrl 修饰符">`^D`</span> EOF | **TTY** 触发 EOF 状态 | 字节 `0x04` 直接传给应用程序 |
| 回显 (Echo) | **TTY** 自动显示输入 | 应用程序决定是否回显 |

Linux 和 BSD 系统提供了 `cfmakeraw()` 便捷函数。在底层，它禁用了 termios 结构中的 `icanon`、`echo`、`ISIG`、`IXON` 等标志位。

### 三个角色

```
终端模拟器 (Ghostty)  —  捕获按键，转换为字节，写入 PTY Master
        ↓
TTY 驱动 (内核)       —  PTY Slave 端，根据 termios 设置拦截或传递字节
        ↓
应用程序 (Fish/Helix/  —  调用 tcsetattr() 切换至 Raw Mode，接管所有按键处理
  Claude Code)
```

## 8. Ctrl+J vs Ctrl+M：深度解析

这两个键的区别揭示了 termios 标志如何影响应用程序看到的内容：

| 按键 | 字节 | 终端默认行为 | 能否独立绑定？ |
|-----|------|-------------|---------------|
| `Ctrl+M` | `0x0D` (CR) | 等同于 Enter | **Cooked Mode** 下不行；在 **Raw Mode** 下配合现代协议才行 |
| `Ctrl+J` | `0x0A` (LF) | 与 Enter 不同 | 可以 —— 字节不同 |

### 在 **Cooked Mode** 下

**TTY** 驱动默认启用 `icrnl` (Input CR to NL) 标志。这会将输入的 `0x0D` (CR) 转换为 `0x0A` (LF)。对于应用程序，`Ctrl+M` 和 `Ctrl+J` 都产生字节 `0x0A` —— 它们是无法区分的。

### 在 **Raw Mode** 下

当应用程序切换到 **Raw Mode** 时，`icrnl` 被禁用。这允许应用程序区分 `Ctrl+J` (`0x0A`) 和 `Enter/Ctrl+M` (`0x0D`)。

然而，一个常见的误区是认为仅靠 **Raw Mode** 就能区分 `Ctrl+M` 和 `Enter`。在标准终端配置中，这两个按键发送的字节完全相同 (`0x0D`)。如果字节完全一样，即使在 **Raw Mode** 下，应用程序也无法区分它们。要真正区分 `Enter` 和 `Ctrl+M`，必须依赖终端模拟器发送不同的转义序列。

## 9. 现代方案：Kitty Keyboard Protocol 与 CSI u

我们难以实现 `Shift+Enter` 或区分 `Tab` 与 `Ctrl+I` 的根本原因是僵化的 1963 年 ASCII 编码。为了在不破坏向后兼容性的情况下解决这个问题，现代终端模拟器（如 Ghostty, Kitty, Alacritty）支持增强的键盘协议（如 Kitty Keyboard Protocol 或 CSI u 标准）。

这些协议不再为 Enter 发送单一的遗留字节 `0x0D` (CR)，而是可以发送丰富的转义序列：

- 遗留 Enter: `\x0D`
- 现代 Shift+Enter (CSI u): `\x1b[13;2u`

应用程序（如果支持这些协议）可以解析这些转义序列，从而为每一个可能的按键组合提供完全独立的键绑定，最终摆脱 **TTY** 时代硬件的约束。

## 10. 调试终端输入：亲眼查看字节

你可以使用标准的 Unix 工具观察各层的终端输入处理。

### 查看当前 **TTY** 设置

```bash
stty -a
```

### 观察原始按键字节

```bash
# 按下按键，然后 Ctrl+C 退出
cat | xxd
```

你会看到每个按键的原始十六进制值。尝试比较 `Ctrl+M` (显示 `0d`) 和 `Ctrl+J` (显示 `0a`)。

### 切换至 **Raw Mode** 观察

```python
import sys, tty, termios

fd = sys.stdin.fileno()
old = termios.tcgetattr(fd)
try:
    tty.setraw(fd)
    while True:
        ch = sys.stdin.read(1)
        print(f"byte: {ord(ch):02x} ({repr(ch)})", file=sys.stderr)
finally:
    termios.tcsetattr(fd, termios.TCSADRAIN, old)
```

## 常见问题

### 为什么我不能独立于 Enter 重新绑定 Ctrl+M？

在 **Cooked Mode** 下，不行：`Ctrl+M` 产生 `0x0D` (CR)，**TTY** 驱动的 `icrnl` 标志将其转换为 `0x0A` (LF) —— 与 `Ctrl+J` 相同。

在 **Raw Mode** 下，`icrnl` 已禁用，但 `Ctrl+M` 和 `Enter` 通常仍发送相同的字节 (`0x0D`)。要区分它们，你需要终端模拟器和应用程序都支持 Kitty Keyboard Protocol 或 CSI u 等现代协议，这些协议会为不同的组合键发送唯一的转义序列。

### Ctrl+J 和 Ctrl+M 有什么区别？

`Ctrl+J` 产生 `0x0A` (换行符)，而 `Ctrl+M` 产生 `0x0D` (回车符) —— 它们是不同的字节。但在 **Cooked Mode** 下，**TTY** 驱动会把 `0x0D` 转换为 `0x0A`。在 **Raw Mode** 下，字节会原样到达，允许独立绑定。

### `stty raw` 到底做了什么？

它禁用了所有 **TTY** 驱动处理：关闭 `icanon` (行编辑)、`echo` (自动显示)、`ISIG` (信号生成)、`IXON` (流控制) 和 `icrnl` (CR 转 NL)。应用程序随后会收到输入的每一个原始字节。

### 为什么 `Ctrl+U` 是删至行首而非整行？

在 POSIX termios 中，`VKILL` (`Ctrl+U`) 定义为“删除整行”。但当 <span title="Bourne Again SHell — 大多数 Linux 发行版的默认 shell">Bash</span> 或 <span title="Friendly Interactive SHell — 用户友好的交互式 shell">Fish</span> 切换到 **Raw Mode** 时，**TTY** 处理被绕过，应用程序将 `Ctrl+U` 重新定义为“删除至行首”（保留光标后的字符），这在实践中更实用。

## 结论

终端键盘输入处理是一个三层系统：ASCII 定义了字节级编码（自 1963 年未变），**TTY** 驱动提供了源自 **TTY** 设计的行编辑和信号处理，而应用程序层处理其余一切。

现代终端工具中的键绑定 —— `Ctrl+A` 行首、`Ctrl+E` 行尾、`Ctrl+K` 向后删除 —— 源自 40 年前的 Emacs 惯例，而非 POSIX 或任何正式标准。**Raw Mode** 下的应用程序绕过了 **TTY** 驱动，但出于设计考虑保留了其语义。
